# Економіка Характеристик — Спек 6b: Церкви (дизайн)

**Дата:** 2026-07-12
**Статус:** затверджено до реалізації
**Парасольковий роадмап:** [2026-06-26-ingredient-economy-roadmap.md](2026-06-26-ingredient-economy-roadmap.md)

## Огляд і межі

Спек 6 роадмапу (D/E — перерозподіл) декомпозовано: D (підпільний ринок) реалізовано
([спек 6a](2026-07-10-ingredient-economy-market-design.md)). Частина E розбита на два спеки:

- **цей спек — 6b: церкви** + спільний реєстр УСІХ інституцій (церкви й таємні
  організації — як дані);
- **6c — таємні організації** (наступний спек): предмет-ключ вступу, завдання організацій
  (включно з рейдами на сховища церков, убивством священиків), шпигунство/викриття,
  механіка Якоря.

Цей спек додає:

- **реєстр інституцій** — кодовий, чистий домен: 10 церков + ~25 таємних організацій із
  канонічними шляхами й рівнями доступу (повний / неповний до N-ї послідовності);
- **членство в церкві**: вступ у храмі, вклад (очки), ранги, вихід із ціною;
- **храми**: адмінська прив'язка місця (`/church bind`) + Citizens-NPC-священик як єдина
  точка взаємодії (меню лише через NPC);
- **службу**: ротаційні завдання (полювання, доставка) + постійні доручення (зупинити
  рампейджера, пожертви);
- **крамницю за очки вкладу** (інгредієнти й книги рецептів домену церкви) і **сховище
  Характеристик** із реальних пожертв (ціль рейдів у 6c).

**Поза межами:** уся механіка таємних організацій (6c); запечатані артефакти (не
реалізовані в плагіні); Якір для Seq 2 (згадується як мета нейтральних організацій — 6c+);
генерація унікальних храмових структур; шпигунство/викриття подвійних агентів (6c).

## Ухвалені рішення (з брейнштормінгу)

| Питання | Рішення |
|---|---|
| Церкви vs таємні організації | Різні інституції: церква = віра в домен, служіння, капітал, ранги за вклад; організація = конкретні цілі, «завдання → нагорода» (6c) |
| Ростер | Усі канонічні інституції закладаються наперед у кодовий реєстр; шляхи, яких ще немає в плагіні, «оживають» з появою шляху |
| Присутність | Церкви — фізичні храми з NPC; таємні організації — без адреси (вступ через предмет-ключ, 6c) |
| Членство | Одна церква + (у 6c) одна таємна організація паралельно — шпигунство дозволене задумом |
| Завдання церков | Полювання на ворожих істот, зупинка бушуючих + спільні прості (доставка, пожертви) |
| Нагороди | Церква: вклад → ранги → крамниця за очки; організація (6c): пряма видача |
| Характеристики | Церква НЕ емітує: сховище наповнюється лише пожертвами членів; викуп — Кардинал, свого шляху. Інваріант роадмапу цілий |
| Архітектура | Кодовий реєстр + патерн ринку (чистий домен + application-оркестратор + JSON-персистентність); членство в окремому файлі, не в `Beyonder` |

## Реєстр інституцій — `domain.organizations` (нуль Bukkit)

- **`Institution`**: `id` (kebab: `church-evernight`, `order-aurora`), `type`
  (`CHURCH` / `SECRET_ORDER`), українська назва, лор-рядок (божество/ідея), список
  `PathwayAccess`. **Порожній список доступів у `SECRET_ORDER`** = «шляхи залежать від
  учасників» (приймає будь-який шлях); у `CHURCH` порожнім не буває (тест).
- **`PathwayAccess`**: канонічна англійська назва шляху (ключ `PathwayManager`: `"Fool"`,
  `"Door"`, `"WhiteTower"`; для нереалізованих — `"Darkness"`, `"Tyrant"`, …), рівень
  **FULL** або **PARTIAL(minSequence)** («неповний до N-ї послідовності»: інституція
  підтримує шлях лише до Seq N включно — товари/рецепти сильніші за N недоступні),
  опційна помітка гілки (для Церкви Блазня: «Сім'я Авраам», «Місячне Місто», …).
- **Правило пінінгу неоднозначностей канону:** «повний або неповний» → беремо FULL;
  «щонайменше до N-ї» → PARTIAL(N). Точна таблиця — у коді реєстру.
- **`InstitutionRegistry`** — статичні дані в коді (як `PathwayManager.initializePathways`).

## Членство, вклад, ранги

- **`Membership`** (домен): `playerId`, `institutionId`, **`lifetimeContribution`**
  (сумарний вклад, не зменшується — визначає ранг) і **`balance`** (поточні очки —
  валюта крамниці, витрачаються при купівлі). Модель тримає щонайбільше **одну церкву**
  на гравця + окремий слот під одну таємну організацію (заповнюється в 6c).
- **Вступ** (чисте правило): церква приймає, якщо гравець (а) без шляху, або (б) його
  шлях є в доступах церкви. PARTIAL не блокує вступ — лише обмежує стелю крамниці.
- **Ранги** — єдина 5-щаблева драбина: **Вірянин → Служка → Диякон → Єпископ →
  Кардинал**; пороги `lifetimeContribution` у `config.yml` (`church.rank-thresholds`,
  дефолти в коді). Стеля крамниці за рангом: Вірянин Seq 9–8, Служка 7–6, Диякон 5–4,
  Єпископ 3–2, Кардинал 1–0; фактична стеля позиції =
  min(стеля рангу, PARTIAL-ліміт шляху позиції).
- **Вихід** `/church leave`: членство видаляється, вклад і баланс згорають; кулдаун на
  будь-який новий вступ `church.rejoin-cooldown-days` (дефолт 3), персиститься.
- **Персистентність**: `memberships.json` — `JSONMembershipRepository` (патерн
  `GatheringSnapshotRepository`: синхронний запис після кожної мутації; обсяг малий).
  Разом із членством зберігаються активні завдання та кулдаун повторного вступу.

## Храми і NPC-священик

- **Прив'язка, не генерація.** Будівлю адмін зводить як завгодно (вручну /
  `/structure`); `/church bind <institutionId>` у точці стояння → запис у
  `church-sites.json` (`ChurchSiteRepository`) + спавн NPC-священика. `/church unbind` —
  зняти найближчий сайт. Одна церква може мати **кілька храмів**, кожен зі своїм NPC.
- **NPC** — Citizens-трейт `ChurchPriestTrait` (патерн `OrganizerNpcService`): ім'я з
  реєстру («Жрець Вечірньої Богині»), спавн при bind і на старті з `church-sites.json`,
  деспавн при unbind.
- **Клік по священику:** не член → лор-привітання + **[Вступити]** (відмова з причиною:
  чужий шлях / кулдаун); член цієї церкви → меню (triumph-gui, патерн `MarketMenu`):
  **Завдання / Крамниця / Пожертви / Мій ранг**; член іншої церкви → лор-відмова.
- Стиль реплік — як Посередник ринку (action bar / чат, українською).

## Служба (вклад)

**А. Ротаційні завдання** — пул оновлюється раз на `church.tasks.refresh-hours`
(дефолт 24); одночасно активних до `church.tasks.max-active` (дефолт 2); refresh замінює
невиконані:

- **«Вполювати істоту»** — вбити N міфічних істот конкретного id; ціль — істоти **чужих**
  до церкви `PathwayGroup` (мапу «шлях→група» передає application; чистий
  `ChurchTaskGenerator` обирає ціль і масштабує N та нагороду від tier). Зарахування —
  `EntityDeathEvent` + `MythicCreatureGateway.creatureId()`.
- **«Принести пожертву»** — доставити N конкретних інгредієнтів (із рецептів домену
  церкви через `IngredientSequenceIndex`); здача кнопкою в меню Завдань, предмети
  знімаються атомарно (патерн `removeMatching` ринку).

**Б. Постійні доручення** (діють завжди):

- **«Зупинити бушуючого»** — вбивство гравця в стані рампейджу (перевірка
  `RampageManager`) → великий вклад. У 6c розшириться на «бушуючий член ворожої
  організації».
- **Пожертви** (вкладка меню): монети (`CurrencyCodec`; **грошовий сток** — монети
  згорають, як комісія ринку), інгредієнти за таблицею вкладу, **Характеристика шляху
  церкви** → дуже великий вклад + предмет лягає у сховище церкви.

Правила генерації й масштаб нагород — чисті (юніт-тести: складніша ціль → більший вклад;
без дублів у пулі). Прогрес — у `memberships.json`.

## Крамниця і сховище

**Крамниця** (валюта — `balance`):

- Асортимент — лише домен церкви: **допоміжні інгредієнти** рецептів її шляхів
  (`IngredientSequenceIndex`) і **книги рецептів** її шляхів (`recipe:<path>:<seq>`,
  наявна фабрика). Гравець без шляху бачить книги Seq 9 шляхів церкви (онбординг у домен)
  і дешеві інгредієнти.
- Гейт позиції: `seq ≥ стеля` (ранг × PARTIAL). **Готових зілль немає** (інваріант
  ринку: просування не купується напряму).
- Ціни — мапи `seq→очки` у `config.yml` (`church.shop.*`) з дефолтами в коді
  (патерн `MarketConfig.loadSeqMap`).

**Сховище (`ChurchVault`)** — пер-церковне, у `churches-state.json`
(`ChurchStateRepository`):

- Наповнюється **лише пожертвами** Характеристик від членів — канал вилучення з обігу;
  церква нічого не емітує (інваріант роадмапу цілий).
- Вміст — `characteristic:<pathway>:<seq> → кількість` (реконструкція через
  `CharacteristicCodec`, Base64 не потрібен).
- **Викуп:** лише ранг **Кардинал**, лише Характеристика **свого шляху**, за великі очки
  (`church.vault.redeem-points-by-seq`) — чиста рециркуляція дефіциту.
- У 6c сховище стає ціллю завдань-викрадень таємних організацій.

## Wiring, конфіг, команди

- **`ServiceContainer`** (порядок): `InstitutionRegistry` → `ChurchConfig` (`church.*`) →
  `JSONMembershipRepository`, `ChurchSiteRepository`, `ChurchStateRepository` →
  **`ChurchService`** (вступ/вихід, вклад, ранги, крамниця, сховище, пожертви; залежить
  від `BeyonderService`, `CustomItemService`, `IngredientSequenceIndex`, `CurrencyCodec`,
  `MythicCreatureGateway`, `RampageManager`) → `ChurchMenu` → `ChurchPriestTrait`.
  `ChurchListener` (kill-прогрес, рампейдж) — у `registerEvents()`.
- **Команди** (`plugin.yml`): `/church bind <id>` / `/church unbind` — адмінські
  (перевірка `mysteriesabove.admin` у коді, як `/gathering start`); `/church leave`,
  `/church info` — гравецькі. Меню — **лише через NPC у храмі** (фолбек-команди меню
  свідомо нема — фізичність є фічею).
- **`config.yml` `church.*`**: `rank-thresholds`, `rejoin-cooldown-days`,
  `tasks.refresh-hours`, `tasks.max-active`, `shop.ingredient-points-by-seq`,
  `shop.recipe-points-by-seq`, `vault.redeem-points-by-seq`,
  `donation.ingredient-points-by-seq`, `donation.characteristic-points-by-seq`,
  `donation.points-per-coppet`. Усі мапи — з дефолтами в коді (Bukkit не перезаписує
  наявний config.yml).

## Тестування

- **Юніт (чистий домен):** реєстр — id унікальні, доступи валідні (FULL xor
  PARTIAL(0..9)), у `CHURCH` доступи непорожні, кожен шлях із `PathwayManager` покритий
  хоч однією інституцією; правила вступу (без шляху / свій шлях / чужий шлях / кулдаун);
  ранги — пороги, стелі крамниці, min(ранг, PARTIAL); `ChurchTaskGenerator` — масштаб
  нагород від складності, без дублів, refresh замінює пул; сховище — викуп лише
  Кардиналом і лише свого шляху; **інваріант збереження**: пожертвувана Характеристика
  має рівно один вихід (лежить у сховищі або видана викупом).
- **ArchUnit:** `domain.organizations` → до `PURE_DOMAIN`.
- **In-server чекліст:** bind → NPC з'явився → вступ (усі три гілки відмов) → ротаційні
  завдання (kill/доставка) → пожертви (монети/інгредієнт/Характеристика) → пороги рангу →
  крамниця (гейти стелі) → викуп Кардиналом → leave (згорання, кулдаун) → рестарт
  сервера (сайти, членства, сховища відновлені).

## Перевикористання (не переписуємо)

- `MarketConfig`/`loadSeqMap` — патерн конфігу з дефолтами.
- `IngredientSequenceIndex` — послідовності інгредієнтів для крамниці/завдань.
- `CurrencyCodec` — пожертви монетами (сток).
- `CharacteristicCodec` — пожертви/викуп Характеристик.
- `RecipeUnlockService` + фабрика книг рецептів — товар крамниці.
- `MythicCreatureGateway` — ідентифікація істот у kill-завданнях.
- `RampageManager` — доручення «Зупинити бушуючого».
- `OrganizerNpcService`/Citizens-трейти — патерн NPC.
- `MarketMenu`/triumph-gui, `ConfirmationMenu` — патерн меню (викуп Характеристики й
  пожертва Характеристики — через підтвердження).
- `GatheringSnapshotRepository` — патерн JSON-персистентності.
