# Економіка Характеристик — Спек 6b: Церкви (дизайн)

**Дата:** 2026-07-12
**Статус:** затверджено до реалізації
**Парасольковий роадмап:** [2026-06-26-ingredient-economy-roadmap.md](2026-06-26-ingredient-economy-roadmap.md)

## Огляд і межі

Спек 6 роадмапу (D/E — перерозподіл) декомпозовано: D (підпільний ринок) реалізовано
([спек 6a](2026-07-10-ingredient-economy-market-design.md)). Частина E розбита на два спеки:

- **цей спек — 6b: церкви** + спільний реєстр УСІХ інституцій (церкви й таємні
  організації — як дані);
- **6c — таємні організації** (наступний спек): предмет-ключ вступу, завдання організацій
  (включно з рейдами на сховища церков, убивством священиків), шпигунство/викриття,
  механіка Якоря.

Цей спек додає:

- **реєстр інституцій** — кодовий, чистий домен: 10 церков + ~25 таємних організацій із
  канонічними шляхами й рівнями доступу (повний / неповний до N-ї послідовності);
- **автогенерацію храмів біля сіл**: знайдене село → поряд спавниться випадкова ще не
  розміщена церква зі своєю унікальною будівлею; кожна церква існує в світі щонайбільше
  один раз, поки не вичерпаються всі 10;
- **членство в церкві**: вступ у храмі, вклад (очки), ранги, вихід із ціною;
- **службу**: ротаційні завдання (полювання, доставка у сховище) + постійні доручення
  (зупинити рампейджера, пожертви);
- **замовлення готового зілля за очки вкладу** (крамниці немає): церква варить зілля з
  ресурсів власного сховища із затримкою, NPC видає за сповіщенням;
- **сховище церкви**, наперед наповнене інгредієнтами, рецептами й Характеристиками її
  шляхів; поповнюється пожертвами членів; ціль рейдів у 6c.

**Поза межами:** уся механіка таємних організацій (6c); запечатані артефакти (не
реалізовані в плагіні); Якір для Seq 2 (мета нейтральних організацій — 6c+);
шпигунство/викриття подвійних агентів (6c); датапак великих сіл (серверний контент поза
плагіном — механіка реагує на будь-які села, ванільні чи датапачні).

## Ухвалені рішення (з брейнштормінгу)

| Питання | Рішення |
|---|---|
| Церкви vs таємні організації | Різні інституції: церква = віра в домен, служіння, капітал, ранги за вклад; організація = конкретні цілі, «завдання → нагорода» (6c) |
| Ростер | Усі канонічні інституції закладаються наперед у кодовий реєстр; шляхи, яких ще немає в плагіні, «оживають» з появою шляху |
| Присутність | Церкви — фізичні храми (автогенерація біля сіл + адмінський bind); таємні організації — без адреси (вступ через предмет-ключ, 6c) |
| Генерація храмів | Біля знайденого села спавниться випадкова ще не розміщена церква; кожна з 10 — щонайбільше раз на світ; у кожної своя унікальна будівля-структура |
| Членство | Одна церква + (у 6c) одна таємна організація паралельно — шпигунство дозволене задумом |
| Завдання церков | Полювання на ворожих істот, зупинка бушуючих + доставка у сховище (подвійний вклад: за місію і за принесене) |
| Нагорода за очки | ЛИШЕ замовлення готового зілля (крамниці інгредієнтів/книг немає) — довга прогресія |
| Зілля | Вариться з ресурсів сховища (рецепт + інгредієнти або Характеристика-заміна) із затримкою; ресурси списуються |
| Сховище | Наперед наповнене (одноразовий сід — свідомий виняток емісії: багатство церков) + пожертви членів; сирі Характеристики гравцям НЕ видаються (лише згоряють у варінні або крадуться в 6c) |
| Архітектура | Кодовий реєстр + патерн ринку (чистий домен + application-оркестратор + JSON-персистентність); членство в окремому файлі, не в `Beyonder` |

## Реєстр інституцій — `domain.organizations` (нуль Bukkit)

- **`Institution`**: `id` (kebab: `church-evernight`, `order-aurora`), `type`
  (`CHURCH` / `SECRET_ORDER`), українська назва, лор-рядок (божество/ідея), список
  `PathwayAccess`. **Порожній список доступів у `SECRET_ORDER`** = «шляхи залежать від
  учасників» (приймає будь-який шлях); у `CHURCH` порожнім не буває (тест).
- **`PathwayAccess`**: канонічна англійська назва шляху (ключ `PathwayManager`: `"Fool"`,
  `"Door"`, `"WhiteTower"`; для нереалізованих — `"Darkness"`, `"Tyrant"`, …), рівень
  **FULL** або **PARTIAL(minSequence)** («неповний до N-ї послідовності»: інституція
  підтримує шлях лише до Seq N включно — зілля/ресурси сильніші за N недоступні),
  опційна помітка гілки (для Церкви Блазня: «Сім'я Авраам», «Місячне Місто», …).
- **Правило пінінгу неоднозначностей канону:** «повний або неповний» → беремо FULL;
  «щонайменше до N-ї» → PARTIAL(N). Точна таблиця — у коді реєстру.
- **`InstitutionRegistry`** — статичні дані в коді (як `PathwayManager.initializePathways`).

## Членство, вклад, ранги

- **`Membership`** (домен): `playerId`, `institutionId`, **`lifetimeContribution`**
  (сумарний вклад, не зменшується — визначає ранг) і **`balance`** (поточні очки —
  витрачаються на замовлення зілля). Модель тримає щонайбільше **одну церкву**
  на гравця + окремий слот під одну таємну організацію (заповнюється в 6c).
- **Вступ** (чисте правило): церква приймає, якщо гравець (а) без шляху, або (б) його
  шлях є в доступах церкви. PARTIAL не блокує вступ — лише обмежує стелю зілля.
- **Ранги** — єдина 5-щаблева драбина: **Вірянин → Служка → Диякон → Єпископ →
  Кардинал**; пороги `lifetimeContribution` у `config.yml` (`church.rank-thresholds`,
  дефолти в коді). Стеля послідовності замовлюваного зілля за рангом: Вірянин Seq 9–8,
  Служка 7–6, Диякон 5–4, Єпископ 3–2, Кардинал 1–0; фактична стеля =
  min(стеля рангу, PARTIAL-ліміт шляху).
- **Вихід** `/church leave`: членство видаляється, вклад, баланс і невидані замовлення
  згорають; кулдаун на будь-який новий вступ `church.rejoin-cooldown-days` (дефолт 3),
  персиститься.
- **Персистентність**: `memberships.json` — `JSONMembershipRepository` (патерн
  `GatheringSnapshotRepository`: синхронний запис після кожної мутації; обсяг малий).
  Разом із членством зберігаються активні завдання, активне замовлення зілля та кулдаун
  повторного вступу.

## Храми: автогенерація біля сіл + NPC-священик

- **Тригер — знайдене село.** Лістенер завантаження чанків читає структурні старти
  (`Chunk#getStructures`, тип village); ключ села — позиція старту структури. Нове село
  (нема в реєстрі оброблених) і лишились нерозміщені церкви → обирається **випадкова ще
  не розміщена** церква → її **унікальна будівля** ставиться поруч із селом (офсет від
  межі села + прив'язка до рельєфу, дистанція в `church.spawn.village-offset`) → сайт
  реєструється, спавниться NPC. Кожна церква — **щонайбільше раз на світ**; коли всі 10
  розміщені, нові села лишаються без храмів. Оброблені села й розміщені церкви — у
  `church-sites.json` (повторна обробка неможлива після рестарту).
- **Будівлі** — 10 унікальних храмових структур у `mysteries-datapack`
  (`ma:church_<id>`); код ставить їх через Bukkit `StructureManager` (той самий механізм,
  що `/structure`).
- **Адмінський фолбек:** `/church bind <institutionId>` у точці стояння — ручна прив'язка
  (без будівлі; для церков поза лімітом автоспавну або кастомних локацій), `/church
  unbind` — зняти найближчий сайт (працює і для автогенерованих).
- **NPC** — Citizens-трейт `ChurchPriestTrait` (патерн `OrganizerNpcService`): ім'я з
  реєстру («Жрець Вечірньої Богині»), спавн при розміщенні/bind і на старті з
  `church-sites.json`, деспавн при unbind.
- **Клік по священику:** не член → лор-привітання + **[Вступити]** (відмова з причиною:
  чужий шлях / кулдаун); член цієї церкви → меню (triumph-gui, патерн `MarketMenu`):
  **Завдання / Замовлення зілля / Пожертви / Мій ранг**; член іншої церкви → лор-відмова.
- Стиль реплік — як Посередник ринку (action bar / чат, українською).

## Служба (вклад)

**А. Ротаційні завдання** — пул оновлюється раз на `church.tasks.refresh-hours`
(дефолт 24); одночасно активних до `church.tasks.max-active` (дефолт 2); refresh замінює
невиконані:

- **«Вполювати істоту»** — вбити N міфічних істот конкретного id; ціль — істоти **чужих**
  до церкви `PathwayGroup` (мапу «шлях→група» передає application; чистий
  `ChurchTaskGenerator` обирає ціль і масштабує N та нагороду від tier). Зарахування —
  `EntityDeathEvent` + `MythicCreatureGateway.creatureId()`.
- **«Принести у сховище»** — доставити N конкретних інгредієнтів (із рецептів домену
  церкви через `IngredientSequenceIndex`); здача кнопкою в меню Завдань, предмети
  знімаються атомарно (патерн `removeMatching` ринку) і **лягають у сховище церкви**.
  Вклад **подвійний**: за виконання місії + за принесені предмети за таблицею пожертв.

**Б. Постійні доручення** (діють завжди):

- **«Зупинити бушуючого»** — вбивство гравця в стані рампейджу (перевірка
  `RampageManager`) → великий вклад. У 6c розшириться на «бушуючий член ворожої
  організації».
- **Пожертви** (вкладка меню, будь-коли): монети (`CurrencyCodec`; **грошовий сток** —
  монети згорають, як комісія ринку), інгредієнти й книги рецептів → у сховище,
  **Характеристика шляху церкви** → дуже великий вклад + у сховище.

Правила генерації й масштаб нагород — чисті (юніт-тести: складніша ціль → більший вклад;
без дублів у пулі). Прогрес — у `memberships.json`.

## Замовлення зілля і сховище

**Замовлення зілля** — єдине, на що витрачаються очки (`balance`); крамниці
інгредієнтів/книг немає — прогресія свідомо довга:

- **Хто що може замовити:** член із шляхом — лише зілля **наступної послідовності свого
  шляху**; член без шляху — зілля **Seq 9 будь-якого шляху церкви** (онбординг у домен).
  Гейт: seq зілля ≥ min(стеля рангу, PARTIAL-ліміт шляху).
- **Вариться зі сховища:** замовлення можливе, лише якщо сховище містить рецепт цього
  зілля **і** його інгредієнти (за моделлю Спеку 1: усі основні + допоміжні, АБО
  Характеристика[шлях, seq] як заміна всіх основних + допоміжні). Ресурси списуються при
  прийнятті замовлення; бракує — священик каже, чого саме (підказка для пожертв/доставок).
- **Затримка:** `church.order.brew-hours` (дефолт 12, реальний час); по завершенні —
  сповіщення в чат; зілля видає NPC при кліку (замовлення персистується, переживає
  рестарти; одне активне замовлення на гравця).
- **Ціна:** мапа `seq→очки` (`church.order.points-by-seq`), дефолти в коді.

**Сховище (`ChurchVault`)** — пер-церковне, у `churches-state.json`
(`ChurchStateRepository`); вміст — мапа `itemKey → кількість`
(`custom:<id>` інгредієнти, `recipe:<path>:<seq>`, `characteristic:<path>:<seq>`;
реконструкція предметів наявними кодеками, Base64 не потрібен):

- **Початковий сід** при розміщенні церкви: великий запас інгредієнтів рецептів її шляхів,
  книги рецептів усіх доступних їй послідовностей, кілька Характеристик її шляхів
  (кількості — `church.vault.seed.*`, дефолти в коді). Це **свідомий одноразовий виняток
  з інваріанта емісії роадмапу** (лор: церкви накопичували багатство століттями) —
  зафіксувати в роадмапі приміткою.
- **Стримувач емісії:** сирі Характеристики зі сховища гравцям у 6b НЕ видаються ніколи —
  вони або **згоряють у варінні** замовленого зілля (заміна основних інгредієнтів), або
  будуть **украдені** в 6c. Єдиний вихід ресурсів сховища назовні — готове зілля.
- **Поповнення:** доставки-місії та пожертви членів (див. Службу).
- **Інваріант збереження (юніт-тест):** кожна одиниця сховища має рівно один вихід —
  списана у варіння або лежить у сховищі (у 6c додасться «украдена»).

## Wiring, конфіг, команди

- **`ServiceContainer`** (порядок): `InstitutionRegistry` → `ChurchConfig` (`church.*`) →
  `JSONMembershipRepository`, `ChurchSiteRepository`, `ChurchStateRepository` →
  **`ChurchService`** (вступ/вихід, вклад, ранги, замовлення зілля, сховище, пожертви;
  залежить від `BeyonderService`, `PotionManager` (рецепти/варіння), `CustomItemService`,
  `IngredientSequenceIndex`, `CurrencyCodec`, `MythicCreatureGateway`, `RampageManager`) →
  `ChurchMenu` → `ChurchPriestTrait`. `ChurchListener` (kill-прогрес, рампейдж) і
  `ChurchSpawnListener` (чанки/села) — у `registerEvents()`; тикер дозрівання замовлень —
  у `startSchedulers()`.
- **Команди** (`plugin.yml`): `/church bind <id>` / `/church unbind` — адмінські
  (перевірка `mysteriesabove.admin` у коді, як `/gathering start`); `/church leave`,
  `/church info` — гравецькі. Меню — **лише через NPC у храмі** (фолбек-команди меню
  свідомо нема — фізичність є фічею).
- **`config.yml` `church.*`**: `rank-thresholds`, `rejoin-cooldown-days`,
  `tasks.refresh-hours`, `tasks.max-active`, `order.brew-hours`, `order.points-by-seq`,
  `vault.seed.*` (кількості сіду), `donation.ingredient-points-by-seq`,

## Тестування

- **Юніт (чистий домен):** реєстр — id унікальні, доступи валідні (FULL xor
  PARTIAL(0..9)), у `CHURCH` доступи непорожні, кожен шлях із `PathwayManager` покритий
  хоч однією інституцією; правила вступу (без шляху / свій шлях / чужий шлях / кулдаун);
  ранги — пороги, стелі, min(ранг, PARTIAL); `ChurchTaskGenerator` — масштаб нагород від
  складності, без дублів, refresh замінює пул; замовлення — гейти (наступна seq свого
  шляху / Seq 9 без шляху), списання ресурсів за моделлю Спеку 1 (класика АБО
  Характеристика-заміна), відмова з переліком браку; вибір церкви для села — випадковий
  серед нерозміщених, без повторів, стоп після 10; **інваріант збереження сховища**
  (сід + пожертви = у сховищі + списано у варіння).
- **ArchUnit:** `domain.organizations` → до `PURE_DOMAIN`.
- **In-server чекліст:** нове село → храм заспавнився (унікальна будівля, NPC), друге
  село → інша церква; вступ (усі три гілки відмов) → завдання (kill/доставка, подвійний
  вклад) → пожертви (монети/інгредієнт/Характеристика) → пороги рангу → замовлення зілля
  (брак ресурсів → підказка; успіх → затримка → сповіщення → видача) → leave (згорання,
  кулдаун) → рестарт сервера (сайти, оброблені села, членства, сховища, замовлення
  відновлені).
