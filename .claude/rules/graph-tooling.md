# Граф коду (graphify) і його ремонт

Граф у `.graphify/` — індекс для ЗАПИТІВ, не документ для читання. Для питань про
архітектуру спершу `graphify query "<питання>"` / `path` / `explain` / `review-delta`:
вони віддають підграф на кількасот токенів замість грепу по всьому репозиторію.
`GRAPH_REPORT.md` цілком читати не треба — це найдорожчий спосіб дізнатись найменше.
`.graphify/` у `.gitignore` і в git не потрапляє.

## Дві вади, через які граф тихо бреше

1. **Привиди.** `graphify update` (і git-хук, що кличе те саме) прибирає ноди лише тих
   файлів, які зникли ЦІЛКОМ (`semantic_cleanup: dropped N stale node(s) — source file(s)
   no longer present`). Метод, видалений із живого файлу, лишається в графі назавжди зі
   старим номером рядка.
2. **Фільтр «secret».** graphify мовчки викидає будь-який шлях із підрядком `secret`
   (евристика для паролів/ключів). У цьому репозиторії під неї потрапляє
   `SecretOrderService.java` — хаб підсистеми таємних орденів і єдиний із ~490 java-файлів,
   відсутній у графі. Доведено експериментом: із чотирьох ІДЕНТИЧНИХ копій в одній теці
   просканувалась лише та, що без «Secret» в імені (не розмір і не збій парсингу —
   `MarionettistControl.java` на 105 КБ витягується нормально). Вимкнути не можна:
   `--exclude` лише ДОДАЄ патерни. **Повна перезбірка це НЕ лікує — вона викине файл знову.**

Наслідок вади 2 підступний: запит про ордени повертає `OrderMenu`/`OrderCommand`/
`OrderListener` і виглядає повним, хоча хаба, що зв'язує `OrderItems`, `OrderConfig`,
`JSONOrderMembershipRepository`, у ньому немає.

## `scripts/graph-repair.js`

Лагодить обидві вади. Нуль токенів LLM — усе на AST-екстракторі.

```bash
node scripts/graph-repair.js                 # файли останнього коміту + УСІ secret-файли
node scripts/graph-repair.js --since HEAD~5  # усе, що змінилось від ревізії
node scripts/graph-repair.js --files a.java,b.java
node scripts/graph-repair.js --all           # усі .java (повільно, без LLM)
node scripts/graph-repair.js --dry-run       # лише план
```

Secret-файли додаються в набір ЗАВЖДИ, навіть незмінені: вони випадають із графа при
кожній перезбірці, тож потребують ремонту після будь-якого прогону graphify.

Механізм: цільові файли стейджаться в темп **зі структурою тек** (id ноди =
`<тека>_<файл>_<символ>`, тож плаский прогін дав би інші id і дублі замість заміни);
файл із «secret» їде під псевдонімом у ТІЙ САМІЙ теці, а `source_file`, префікс id і label
переписуються назад. Далі ноди цільових файлів у графі ЗАМІНЮЮТЬСЯ свіжими (це й прибирає
привидів), спільноти вже відомих нод переносяться, і в кінці `graphify cluster-only`
розкидає нові ноди по спільнотах, не стираючи наявних назв. Бекап — `.graphify/graph.json.bak`.

## Інваріанти

- Ідемпотентність: повторний прогін дає ті самі числа (викинуто == додано, лічильник після
  `cluster-only` не змінюється). Якщо стрибає — щось зламалось.
- Ребра дедуплікуються по НЕВПОРЯДКОВАНІЙ парі вузлів, бо граф `multigraph: false,
  directed: false`: пара тримає рівно одне ребро. Ключ із `relation` створював би паралельні
  ребра, які `cluster-only` згортає — і лічильник стрибав би між прогонами.
- Ремонт ПІДМНОЖИНИ файлів (`--files`) не відтворює крос-файлових ребер до файлів поза
  набором — AST бачить лише застейджене. Скрипт про це попереджає; повний прогін їх повертає.

## Заборони

- ❌ Перейменовувати класи заради обходу фільтра graphify — інструмент підлаштовується під
  код, не навпаки.
- ❌ Вважати `graphify query` вичерпним для видалення символа з усього репозиторію: граф дає
  орієнтацію на вході, фінальну повноту підтверджує `grep`.
- ❌ Робити повний `/graphify` з нуля заради привидів — він коштує токенів на семантичний
  прохід по доках і не лікує вади 2.
